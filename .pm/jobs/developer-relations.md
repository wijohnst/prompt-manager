# Developer Relations Engineer — Growth

## Why Now

`pm build` is nearly shipped. Once it is, there is nothing telling anyone it
exists, no documentation showing how to use it, and no community presence.
Distribution does not happen by accident. This role is what turns a working
CLI into a product people find and use.

## What They Own

- The npm package README — the first thing any user reads
- End-to-end documentation for `pm build`: installation, `pm.toml` format,
  output format, worked examples
- External communication about prompt-manager: where we show up, what we say,
  what the project stands for
- Feedback loops: surfacing what users find confusing back to engineering

## What They Do Not Own

- Product decisions about what `pm` does — that is the founder
- Application code — that is the full-stack engineer
- Release mechanics — that is the devops engineer

## Done in 30 Days

`pm build` is documented end-to-end in the README. A developer who has never
seen the project can install `pm`, author a `pm.toml`, and produce a working
built prompt by following the docs alone.
